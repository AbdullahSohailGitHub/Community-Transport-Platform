My Process

How I take over an existing codebase.

I read before I write. Every time. No exceptions.

01

Codebase Audit Week 1

Clone the repo, map the folder structure, trace the Firestore schema and security rules, run all portals locally, and test every major workflow. You get a written audit report — what's solid, what's broken, what's missing — before a single line of new code is written.

02

Priority Alignment Week 1 End

We review the audit together and agree on what gets fixed first — critical bugs, security gaps, incomplete portals, or UX improvements. You decide the order, I execute.

03

Structured Sprints Week 2+

Small, tested pull requests rather than big rewrites. Each sprint has a clear scope and deliverable. You see progress weekly, not at the end of a long tunnel.

04

Documentation As We Go

Every module completed gets documented. Architecture decisions, data models, API patterns — all written up so the codebase stays maintainable as your team grows.

05

Mobile-Ready Architecture

Since a React Native app is planned, I structure the Firestore schema and API layer to be mobile-compatible from day one — not a retrofit later.
